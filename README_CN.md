# WebDAV-Proxy-Cipher

这个项目主要是对标准的 WebDAV 服务进行代理，提供基于 WebDAV 协议的透明加解密功能。支持通过各大 WebDAV 客户端直接挂载，实现在线播放加密视频、查看加密图片等功能。所有在 WebDAV 挂载下的操作完全透明，系统会自动实现文件资源的加解密。

到目前为止，此项目的算法已经基本确定下来了，用 RC4 和 AES-CTR 更安全。之前的 mix 混淆明文方案作为加密的话安全强度不够，容易被破解（MIX 已被移除）。而 AES-CTR 更加安全，速度最快。RC4 由于使用 Node.js 进行实现，性能会稍微差一些（但是依然非常高效，电视盒子的性能可以跑满 300M 带宽没问题，主要吃单核性能）。

算法基本确定下来了，接下来就可以正常实现业务了，release 版本应该很快就可以和大家见面了。体验版可以用 Docker 进行体验。

特别感谢原项目 [alist-encrypt](https://github.com/traceless/alist-encrypt) 提供的灵感与参考。

## 一、需求背景

目前市面上有很多云盘或自建的 NAS 服务提供 WebDAV 协议支持。许多人喜欢将云盘通过 WebDAV 挂载为本地磁盘，当做个人的影院，随时在线观看视频。

但不论是公有云盘还是其他存储服务，都普遍存在一个问题：敏感或私人资源可能会被平台扫描并删除，相信很多人经历过文件被和谐的噩梦。最简单的避免方案就是加密后上传，但这带来了极大的局限性：传统加密文件无法通过 WebDAV 直接实时在线播放或预览。同时，加密后的文件分享也存在不便。

WebDAV-encrypt 就是为了解决这个问题而生。它可以结合任何标准的 WebDAV 服务器进行使用。在文件通过代理端上传时自动进行加密，下载时自动进行解密。由于使用的是流加密方案，所以可以很轻松地实现在线播放已加密的视频，浏览图片、文件等。只要你的网盘或 NAS 服务支持 WebDAV 协议，都可以完美接入。

关于这个项目的使用场景：对文件安全隐私有强烈需求，防止云盘扫描删除，同时又需要保留实时播放流媒体 and 便捷下载的体验。

## 二、加密的原理

项目的实现比较简单，当初想法的孵化的项目地址在这里：[tlf-encryption](https://github.com/traceless/tlf-encryption)。它有描述 mix 混淆算法的实现，代理服务的实现思路，也有基础版本的代码实现，可以学习参考。 mix 的方式作为混淆文件没有问题，如果当做是加密的方案，那么强度可能不够，通过文件的特征容易被暴力破解。

### 1. HTTP 代理思路

考虑到 WebDAV 协议本质上是基于 HTTP 的扩展，因此实现了一个标准的 HTTP 代理便涵盖了对 WebDAV 的支持。由于加解密逻辑仅针对 Body（消息体）进行处理，请求头（Headers）与响应头（Headers）在大多数情况下无需修改，直接原路透传即可。

**实现思路：**

1. **请求拦截**：解析客户端请求头并透传至目标 WebDAV 服务器，根据业务逻辑判断是否对请求 Body 进行加密处理。
    
2. **响应拦截**：接收服务器返回的响应头并透传回客户端，同时根据业务规则判断是否对响应 Body 进行实时解密。
    

> 这种拦截与分析思路在各主流编程语言中均可实现，而 Node.js 原生提供的 `http` 模块在处理此类流式业务时尤为高效。

### 2. 具体技术实现

1. **httpProxy.js**：这是 HTTP 代理服务器的核心基础实现。它展示了如何对 HTTP Body 流进行拦截并执行加解密操作。得益于 Node.js 精妙的 `Stream` 设计，开发者可以直接利用读取流、写入流以及 `Transform` 转换流接口进行数据加工，其开发效率和代码优雅度极高。
    
2. **app.js**：基于基础代理实现的业务逻辑层。它专门针对 WebDAV 流量进行劫持，利用流式加密（Stream Cipher）技术对上传、下载以及流媒体播放数据进行实时加解密处理。目前默认代理 `/dav/*` 路径用于验证，用户也可根据需求配置代理整个服务（如 ~~Alist~~Openlist）。
    
3. **兼容性验证**：当前的 Node.js 版本已在阿里云盘等主流服务端完成测试。涵盖了上传、删除、移动及下载（已完美解决 302 重定向问题）等核心操作，确保加解密过程准确无误。详细配置请参考 `config.js`。
    

所以当前项目已经新增了 2 个算法，一个是 RC4，一个是 AES-CTR 算法：

1. AES-CTR 是一种流式加密算法，目前的实现已经在项目中。它的速度比 RC4 快，也比 chacha20 速度快。推荐 armV8 以上的 CPU 使用，X86 架构的也推荐在支持 AES 指令的机器使用。
    
2. RC4 安全性也是足够，理论上安全强度还是有点瑕疵，但已经是可以忽略不计了。它适合在 CPU 不支持 AES 指令 bird 的设备中使用，速度也是非常的快。
    

> 目前比较推荐的是 AES-CTR，不管是从性能还是安全都很不错，目前大部分设备的 CPU 都支持 AES 指令. 从未来使用来看，强烈推荐大家使用此算法. 当然 RC4 兼容性更强，算法很简单，性能也不错。

## 三、安装使用

### 下载运行

需要先安装 Node.js 的环境，具体安装方法，请参考网上的教程。

1、下载此项目，进入 node-proxy 目录执行

- 不安装开发所需的依赖（仅运行、节省存储空间）
    
    > `npm i --omit=dev`
    
- 安装全部依赖（开发、编译等）
    
    > `npm i`
    

2、修改 `conf/config.js` 配置文件，添加你的目标 WebDAV 服务地址 and 端口，并配置你需要进行加密的 WebDAV 文件夹路径。

3、然后执行启动命令

> npm run serve

服务启动后，请直接通过 WebDAV 客户端连接 `http://127.0.0.1:5344/dav/` 即可访问经过透明加解密代理的 WebDAV 服务。

### Docker 部署（本地构建）

由于不再提供预构建镜像，请通过项目提供的 `Dockerfile` 进行本地构建部署。`compose.yaml` 示例（请确保与 `src` 目录处于同一层级）：

```
services:
  webdav-encrypt:
    build:
      context: ./src                             # 解压后源码的相对路径
    restart: unless-stopped
    hostname: webdav-encrypt
    container_name: webdav-encrypt
    volumes:
      - ./src/node-proxy/conf:/node-proxy/conf   # 挂载配置文件目录
    environment:
      TZ: Asia/Shanghai
      WEBDAV_HOST: 192.168.31.254:5254           # 设置上游 WebDAV 地址
    ports:
      - 5344:5344
    network_mode: bridge
```

**启动命令：**

```
docker compose up -d --build
```

配置之后，客户端挂载 `http://127.0.0.1:5344/dav/` 即可正常使用。

### 操作使用

1、因为本项目是透明代理，所以你在 WebDAV 客户端上的所有常规操作（新建、重命名、移动等）都会被透传到上游的 WebDAV 服务器上。

2、如果设置了加密的文件夹目录，那么通过代理端口上传的文件就会被加密，直接在上游云盘查看下载时将无法打开（显示为乱码或加密文件）。但是只要你使用挂载了本代理服务的 WebDAV 客户端，就可以正常下载查看、在线播放流媒体视频、查看图片等，体验与未加密时完全一致。

3、界面上对于路径的设置目前支持正则表达式。推荐表达式例如: `movie_encrypt/*` ，这样的话所有的 `movie_encrypt` 目录下的文件都会被加密传输。

### 局限性

- 因为是代理流量，所有数据都会流经本项目部署的环境，因此该环境的网络带宽对最终体验影响较大。建议部署到本地或者局域网设备中。
    
- 由于webdav基础协议不支持文件锁功能，多客户端同时配置双向同步可能会出现旧版本覆盖新版本内容，该问题与本项目无关，需要客户端实现写入前创建锁文件，或使用统一控制平面管理
    
- 考虑到Nodejs的单进程特性，海量数据加解密场景，单服务器加密性能存在一定瓶颈。本项目可以分布式部署，并使用负载均衡器作为入口，但多IP回源可能触发WebDAV服务商风控，请遵守使用条款，封禁清退与本项目无关。
    

## 五、FAQ

### 1、代理服务器性能

目前经过多种算法的测试，CPU: s905l3a，RC4 测试理论是可以跑满 300Mbps 带宽的，AES-CTR 可以跑满 800Mpbs+ 的带宽，所以你完全不用担心的它的性能会出现瓶密。

### 2、会考虑其他语言的实现吗？

暂时不会考虑。因为这个项目比较简单，无论是使用 Go 还是 Node.js 应该都是相当容易实现的，它的业务不会膨胀到需要用 Java 语言来实现。对于性能的要求也不一定要用到 Go 来支持。选 Node.js 的原因是其在 Web 开发效率上非常高。

目前 Node.js 也支持 pkg 打包，项目已经支持打包成跨平台的可执行文件。

当前版本还不具备大规模生产使用的条件，只是基本实现了核心加解密代理功能，还有部分网盘的 WebDAV 兼容性没测试全面，以及一些附加功能需要开发。分布式部署还需要在更多复杂环境测试一番。此项目会持续关注标准 WebDAV 协议的兼容性，努力适配各大常见服务端，也希望大家多多支持，提些建议。
